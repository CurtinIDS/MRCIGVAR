# Contributing to MRCIGVAR

This document records the current development standards for the refactored `MRCIGVAR` package. It is intended to keep future contributions consistent with the package architecture, naming scheme, and documentation style already established in the refactor.

## Scope

`MRCIGVAR` is an econometrics package for VAR, CIVAR, MRVAR, MRCIVAR, GVAR, CIGVAR, MRGVAR, and MRCIGVAR workflows, including:

- data generation and input helpers
- estimation
- model selection
- impulse response analysis
- diagnostics, constraints, and supporting utilities

The package is being maintained as a refactored codebase with improved naming, documentation, and compatibility wrappers for legacy APIs.

## Core principles

Contributions should follow these rules:

- Preserve numerical behaviour unless a change is explicitly intended and justified.
- Prefer small, local edits over broad style rewrites.
- Keep public APIs predictable and consistent across model families.
- Document every exported function with complete roxygen2 metadata.
- Maintain backward compatibility when renaming or reorganising public functions.

For this project, consistency is more important than stylistic perfection.

## Repository-specific rules

Several files are generated or managed by tooling. Do not edit them casually.

- `NAMESPACE` is generated by roxygen2. Do not edit it by hand.
- `README.md` is generated from `README.Rmd`. Edit `README.Rmd`, then rebuild the README.
- Most module files in `R/` include the header:
  `# (re)generated by tools/02_split_to_modules.R`
  Preserve that header. If the split/generation workflow is being used, make changes in the appropriate source/tooling path and regenerate as needed.
- `R/compat-legacy-aliases.R` is auto-generated for backward compatibility.
  Do not edit it by hand.
  It intentionally contains no roxygen blocks.

If a contribution changes generated outputs, include both the source change and the regenerated artifact in the same commit/PR.

User-facing workflow changes should also be reflected in:

- `README.Rmd` when the package overview or entry example changes
- the relevant vignette in `vignettes/` when longer-form usage guidance changes
- `docs/` only for MkDocs-native pages that are not generated from `README.Rmd`
  or `vignettes/*.Rmd`

## Documentation site

The repository now includes an `MkDocs` documentation site for navigable
project documentation.

- `mkdocs.yml` defines the site structure and theme configuration.
- `scripts/render-docs.R` regenerates the MkDocs pages derived from
  `README.Rmd` and selected vignette sources.
- `docs/` contains both generated MkDocs pages and MkDocs-native pages.
- `environment-docs.yml` defines the conda environment for local preview and CI
  builds.
- `.github/workflows/mkdocs.yml` builds and deploys the site to GitHub Pages.

For local docs work:

```bash
conda env create -f environment-docs.yml
conda activate mrcigvar-docs
Rscript scripts/render-docs.R
mkdocs serve
```

Use `mkdocs build --strict` before pushing documentation-heavy changes when
possible.

## File organisation

Keep functions in the module that matches their responsibility:

- `R/data.R`, `R/data_input.R`: data generation and input construction
- `R/estimation.R`: model estimation
- `R/model_selection.R`: lag/threshold/model selection
- `R/irf.R`: impulse response functions and plotting
- `R/diagnostics.R`, `R/constraints.R`: testing and diagnostics
- `R/utils.R`: reusable low-level helpers
- `R/compat-legacy-aliases.R`: generated compatibility wrappers only

Do not place new exported functions in `utils.R` if they belong to a clearer domain module.

## Naming conventions

### Public functions

Use lower snake_case for all new/refactored public functions.

Examples:

- `cigvar_data()`
- `cigvar_estimate()`
- `cigvar_select_()`
- `irf_mrcigvar_cb()`
- `infer_deterministic_type()`

Use suffixes consistently across model families:

- `_data` for model/data constructors
- `_estimate` for estimation routines
- `_select_` or `_select_m` for model-selection helpers where this pattern already exists
- `_cb` for confidence-band variants
- `_rm` for regime-migration GIRF helpers
- `_test` for testing/constraint functions

If an existing family already uses an established suffix pattern, follow that pattern rather than inventing a new one.

### Internal helpers

Internal helpers may be shorter, but should still be descriptive enough to identify their purpose.

Prefer:

- `bo_ao_w2_g()`
- `gw2_bo_ao()`
- `summary_civar()`

Avoid introducing new names that are:

- CamelCase
- inconsistent with neighbouring functions
- cryptic unless they reflect a standard algebraic object already used across the package

### Legacy names

Legacy names are preserved only through generated wrappers in `R/compat-legacy-aliases.R`.

- Do not add hand-written deprecated aliases in other files.
- If a public function must be renamed, update the rename map/tooling that generates compatibility wrappers.

## Function design standards

### API shape

Public functions should use argument names that match the package vocabulary already in use:

- `res` for fitted/model objects
- `n`, `m`, `p`, `T`, `S` for dimensions and regimes
- `W`, `Sigma`, `Sigmao`, `Bo`, `Ao`, `Co`, `Go` for model matrices/arrays
- `irf`, `comb`, `conf`, `runs`, `state` for IRF routines

When extending an existing model family, mirror the argument order and return shape of sibling functions where practical.

### Return values

Return values should be structurally consistent within a function family.

- Estimation functions generally return the input object augmented with fitted results.
- Helper transformations typically return `list(...)` or arrays/matrices.
- Model-selection functions return matrices or lists with clearly ordered criteria.

Do not silently change the type or structure of an established return object without a strong reason.

### Side effects

Prefer pure functions. Avoid:

- hidden writes to global state
- attaching packages
- modifying inputs outside the returned object

## Code style

This codebase mixes legacy numerical style with refactored documentation. Follow these rules:

- Preserve surrounding style inside an existing function unless there is a strong readability reason to clean it up.
- Do not rewrite large blocks only to change spacing or assignment operators.
- Keep edits minimal and reviewable.
- Use explicit namespace qualification where the codebase already does so for external functions, for example `stats::lm()`, `Matrix::nearPD()`, `ggplot2::ggplot()`.

When writing new code:

- prefer readable, explicit variable names when they do not conflict with the established mathematical notation
- keep matrix dimensions and array shapes easy to infer
- add short comments only where the algebra or indexing is otherwise hard to parse

## Documentation standards

The package uses roxygen2 with markdown enabled:

- `Roxygen: list(markdown = TRUE)`

Every exported function must have a complete roxygen block immediately above the function definition.

### Required sections

At minimum, provide:

- title line
- short description paragraph
- `@param` entries for every argument
- `@return`
- `@examples`
- `@export` for public functions

Use `@keywords internal` only when the function is exported for package workflow reasons but is still primarily an internal helper.

### Title style

Use short, imperative or descriptive titles in sentence case.

Prefer:

- `Estimate a cointegrated GVAR model`
- `Select lag specifications for a CIGVAR model`
- `Convert VAR coefficients to error-correction form`

Avoid:

- overly long titles
- titles that repeat the full description
- inconsistent casing

### Description style

Descriptions should answer:

1. What the function does.
2. Which model family or object type it operates on.
3. Any important variation, such as single-regime vs multi-regime behaviour.

Keep the first paragraph concise. Add more detail only if needed for interpretation.

### Parameter style

Parameter descriptions should:

- start with the parameter name already provided by roxygen
- describe the expected type and role
- mention key dimensions when relevant
- mention allowed values for flags/options

Prefer:

- `Integer scalar giving the number of IRF horizons.`
- `Numeric vector of candidate threshold values.`
- `A fitted CIGVAR object, typically returned by \`cigvar_estimate()\`.`

### Return style

Return descriptions should state:

- the object type
- dimensions when relevant
- ordering/meaning of major components

Prefer:

- `A three-dimensional array of dimension (m * n, m * n, nstep).`
- `A list containing domestic coefficient arrays B, foreign coefficient arrays A, and deterministic coefficients C.`

### Examples

Examples should be informative but not fragile.

- Keep runnable examples reasonably small.
- Use `\dontrun{}` for expensive, long, or numerically sensitive examples.
- Match object names and conventions used elsewhere in the package.
- Do not include examples that rely on undefined objects or inconsistent argument names.

## Exports and internal helpers

Only export functions that are part of the user-facing API or are already established as exported helpers.

Before exporting a new function, ask:

- Is this intended for package users?
- Is there already a helper that should remain internal?
- Does the function need a compatibility alias if it replaces an older public name?

If a helper is not meant to be user-facing, prefer leaving it unexported.

## Backward compatibility

Backward compatibility matters in this package.

When changing public functions:

- preserve argument names where possible
- preserve return structure where possible
- avoid breaking old examples or model workflows without a migration path
- use generated legacy aliases when renaming public functions

Do not remove established public functions without discussion and a coordinated compatibility plan.

## Testing and verification

Before merging a substantial contribution, contributors should run the relevant checks they can support locally, such as:

- roxygen regeneration if documentation changed
- package checks
- representative examples for the affected model family
- targeted regression tests for any numerical bug fix

For numerical code, verify both:

- the code runs
- the output shape and interpretation still match the documented API

## Pull request expectations

A good contribution should include:

- a focused change
- updated roxygen documentation if the public API changed
- updated vignette or README content if the user-facing workflow changed
- regenerated package artifacts if required
- a short explanation of any numerical or API tradeoff

If the change is intentionally behaviour-changing, say so explicitly and document:

- what changed
- why it changed
- which model families/functions are affected

## In short

If you are unsure, optimise for:

- API consistency across model families
- documentation quality
- minimal behavioural churn
- compatibility with the refactored naming scheme
- respect for generated files and tooling
